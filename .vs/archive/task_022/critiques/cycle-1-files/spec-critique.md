# Spec Critique — task_022: `vibe audit --history` performance

Plain English, adversarial read of `/workspace/.vs/spec.md`. Default to flagging.

## Concerns

1. **BLOCKING (AC1) — the 10-minute OLD-scanner budget is empirically tight, maybe insufficient, and has no fallback.**
   I measured the current scanner directly: `git log -p --no-color --all` over this repo produces 70,746 lines / 51,806 added (`+`) lines across 228 commits. A 2,000-line slice (1,970 added lines) through `vibe-content-scan.sh --blob-stdin` took 16.85s real. Extrapolating: by added-line rate (0.00855s/line) → ~443s (~7.4 min); by raw-line rate (0.00842s/line) → ~596s (~9.9 min). Either extrapolation lands close to or inside AC1's 10-minute cap, on this sandbox, with no other load. On a slower CI box, or under the concurrency a `/vs` cycle typically runs under (Generator + Tester + other subagents), this could easily exceed 10 minutes. The spec calls the 10-minute figure "acceptable" but never proves it, and AC1 is a **hard gate** with no defined fallback if the old scanner doesn't finish (does the Tester kill it and fail the cycle? extend the timeout? fall back to a slice?). Recommend: either raise the budget with margin, or make AC1 explicitly slice-based (e.g., a bounded N-commit window) rather than depending on full history completing in a fixed wall-clock window.

2. **BLOCKING (Part A / AC1-AC2-AC4) — no acceptance criterion mechanically verifies the mandated implementation shape.** Part A prescribes a specific technique (bash-native `[[ =~ ]]` + `BASH_REMATCH`, no forking) in named functions. Every AC that touches this is purely behavioral: AC1/AC2 check output fidelity, AC4 checks aggregate wall time, AC7 checks only for bash-4+/`\b` constructs. Nothing checks that `check_rule`, `check_email_rule`, `check_home_path_rule`, `is_named_trailer`, `is_trailer_line` actually stopped forking `printf | grep`. A Generator (or a future editor) could pass every AC via a different speedup — e.g. collapsing multiple rule checks into fewer `grep` invocations, or fixing only the highest-cost functions while leaving others forking — and never trigger a test failure as long as fidelity and the 60s target hold today. This is exactly the "trivial test-evasion route" this audit is supposed to catch: the letter of Part A (named technique) isn't enforced, only an outcome that's cheap to satisfy by other means. Recommend extending AC7 (or adding an AC8-adjacent check) to grep the final script for absence of subprocess-forking idioms (`| grep`, `$(printf ... | ...)`) inside the five named functions specifically.

3. **BLOCKING (AC3) — "after location normalisation" is undefined and looks like a loophole.** B1 says `--messages-stdin` emits location as `commit <sha>` directly. The OLD pass 2 rewrites `--message`'s literal `"message"` location to `commit <sha>` via `awk`. Both paths should therefore already produce byte-identical `commit <sha>` strings with no transform needed — same sha source (`%H`), same literal prefix. If that's true, "after location normalisation" is vestigial and should be deleted, with AC3 requiring true byte-identical comparison (consistent with AC1/AC2's standard). If it's NOT true — if some real difference between the two paths' location strings is anticipated (whitespace, truncation, ordering) — the spec must say what it is, because as written a Tester is free to define "normalisation" as generously as needed to paper over a genuine per-commit attribution bug, which is precisely the kind of finding B3 is worried about (misattribution).

4. **BLOCKING (Part A feasibility trap) — `set -euo pipefail` + bare `[[ $content =~ $pattern ]]` is a script-killer if not wrapped.** The script has `set -euo pipefail` active (line 33). A bare `[[ $content =~ $pattern ]]` statement, when the pattern does NOT match (the overwhelmingly common case per line, since most lines trigger zero of the ~9 rule checks), returns 1. Used as a standalone statement (not the condition of `if`/`while`/`&&`/`||`), that non-zero exit would trip `set -e` and terminate the entire scanner on the very first non-matching line of real input — which is most lines. The spec's Part A directive ("bash-native `[[ $content =~ $pattern ]]`") doesn't pin the required `if [[ ... ]]; then ... fi` (or equivalent) wrapping needed to survive `set -e`. This should be an explicit constraint, not left to be inferred, given the Generator is asked to touch a `set -euo pipefail` security-boundary script.

5. **BLOCKING (Part A feasibility trap, named in the audit brief) — nocasematch leakage on early returns is asserted but never tested.** A1 requires `shopt -s nocasematch … shopt -u nocasematch`, "restore unconditionally; never leave it set" — correctly anticipating the risk. But `check_rule` has an early-return path (`[ -z "$m" ] && return 0`) that sits between where nocasematch would be set (for the icase=1 `secret-assignment` rule) and where it would need to be restored. No AC exercises this: nothing in AC2's corpus combines an icase=1 rule evaluation with a subsequent case-sensitive rule check on the *same line*, in the *same `scan_line` call*, to prove nocasematch didn't leak. Concretely: a line containing both a mixed-case `secret-assignment` trigger AND an uppercase MDNS-shaped string (`FOO.LOCAL`, which the case-sensitive `mdns-local` rule, icase=0, must NOT match) would catch this if nocasematch leaked. Recommend adding this exact case to AC2's corpus.

## Minor

6. **AC5's mode list omits `--messages-stdin` itself.** AC4 (output/exit contract) applies scanner-wide, and A4 says exit is "0/1 only," but AC5 — which is where exit-code correctness gets tested on clean/dirty fixtures per mode — lists `--staged`, `--message`, `--blob-stdin`, `--range`, `--identity` and not the new mode. `_audit_history`'s own exit logic doesn't depend on the scanner's exit code for pass 2 (it recomputes from `$tmp_messages` content, per B2's "as today" / `|| true` pattern), so this doesn't threaten `vibe audit` itself, but the new mode's own 0/1 contract in isolation goes untested.

7. **B1's "malformed input (no sha line) must not crash" has zero test coverage anywhere in the ACs.** AC7 is static-only (grep for forbidden constructs); no AC constructs a malformed NUL stream (empty record, truncated record, body-only record) to prove the defensive code actually survives it rather than just existing. Recommend a concrete fixture test for this exact case.

8. **AC1's "sorted finding lines are byte-identical" doesn't pin sort locale/collation.** Cheap to close (`LC_ALL=C sort`) and unlikely to bite since both runs happen in the same shell/locale, but for a byte-identical hard gate it costs nothing to pin explicitly.

9. **AC1's "pre-change-sha" isn't concretely defined.** Should read something like "the commit immediately preceding the Generator's first commit for this task" — matters if a repair cycle happens and someone naively diffs against the wrong baseline.

10. **AC2 doesn't test `--message` or `--range` mode parity directly**, only `--blob-stdin` and `--staged`. The shared-code-path argument (all modes funnel through `scan_line`/`scan_diff_stream`) is reasonable, but `--range` exercises tier `"block"` only, a code path AC2's two tested entry points don't specifically hit at that tier. Worth stating as a deliberate coverage decision rather than leaving it as an implicit gap.

11. **No permanent regression guard against the fork-storm reappearing.** Per the Test-location section, AC1 and AC4 are one-off Tester-log entries (permanent tests "must not depend on the vibe repo's own mutable history"), and AC2's fixture corpus is too small to exercise fork-storm-scale timing. Combined with concern 2 (no implementation-shape check), this means a future edit could silently reintroduce subprocess-forking in these functions and nothing in the permanent suite would catch it until the vibe repo's real history grows large enough to time out again — the exact failure mode this task exists to fix.
