# Spec Critique — task_026 (`vibe pat [repo]` PAT rotation + launch-time 401 reprompt)

## Concerns

1. **(Contract bullet 13 / AC8) "lookup_token success branch" is a term that doesn't map onto the actual function's control flow, and the empty-token case is untested.** `lookup_token` (vibe:313-324) always falls through `awk ... || true` — it has no distinct success/failure return path; the caller decides "found vs not found" purely by testing whether `$GITHUB_TOKEN` is non-empty (vibe:2775-2778). The contract's phrase "the `lookup_token` success branch only" is presumably meant to be read as "only when a non-empty token was found," but as literally written it invites a misimplementation where `stored_token_rejected` is called unconditionally right after `GITHUB_TOKEN=$(lookup_token ...)`, including when the result is empty — i.e. exactly the case that's about to run `setup_token` for the first time anyway. That would fire a needless 401 probe with an empty `Authorization: Bearer` header and could print the misleading "Stored PAT ... was rejected" warning when no PAT was ever stored. No AC exercises this: AC8 only tests the `VIBE_PAT_CHECK=0` opt-out, not "empty-token → `stored_token_rejected` not invoked." This is a real correctness gap, not just wording — add an AC (or tighten the contract sentence to something unambiguous like "only when `$GITHUB_TOKEN` is non-empty after lookup") and cover it with a test.

2. **(AC8) The guard AC is close to untestable as scoped, and its own fallback contradicts its "must be executable" requirement.** The real call site lives past the `VIBE_SOURCE_ONLY=1` early-return guard (vibe:2616) and deep inside the docker/devcontainer launch sequence — i.e. exactly the region `smoke-test.py`'s established `VIBE_SOURCE_ONLY=1`-source convention cannot reach (confirmed against `test_token_helpers`, smoke-test.py:368-398, and the `learning`/`repos`/`audit` dispatch pattern at vibe:2614-2648, none of which touch code after that guard). AC8 acknowledges this by permitting "static assertion that the call site tests the env var first" as a fallback — but a source-text/ordering assertion is not meaningfully different from the "prose" check the same sentence says to avoid; it can pass with the two pieces of code in the right order while never once proving that setting `VIBE_PAT_CHECK=0` actually prevents `stored_token_rejected` from running. Either commit to one testable shape (e.g. require Generator to factor the call-site logic into a small sourceable function like `maybe_check_stored_token <repo> <token>` that smoke-test.py CAN source and drive with a shimmed curl) or accept explicitly that this AC is inherently weaker than the others and drop the "must be an executable test" language.

3. **(Contract bullet 11 / AC4) No pinned exact string for "print the target repo."** The contract gives verbatim strings for the stored/not-stored line, the abort message, the success message, and the warning message — but not for the "print the target repo" component. AC4 needs to verify "it resolves the same slug `detect_github_repo` reports" by reading output, but there's nothing fixed to grep for. Given `test_task026_*` functions are frozen after cycle 1 (Test location, spec.md:50) and Tester is haiku-tier working from the spec alone, this is a real risk of burning the 3-cycle budget on a wording mismatch between what Tester assumes and what Generator writes. Pin the format, e.g. `Repo: <repo>` on its own line before the stored/not-stored line.

4. **(AC10) Gameable presence check.** "presence check by distinctive phrase, e.g. `vibe pat`" is weak enough that a single incidental mention of the literal string `vibe pat` anywhere in README.md and anywhere in MANUAL-TESTS.md satisfies it — without ever documenting the launch-time auto-reprompt behaviour (README) or the `VIBE_PAT_CHECK=0` suppression scenario (MANUAL-TESTS), both of which Files-in-scope explicitly requires. Trivial-evasion example: Generator adds one line "New: `vibe pat`." to README and a one-line MANUAL-TESTS bullet that just says "test `vibe pat`" — AC10 passes, intent missed. Require presence of 2+ distinct phrases per file (e.g. README: `vibe pat` AND `401`; MANUAL-TESTS: `vibe pat` AND `VIBE_PAT_CHECK=0`).

5. **(Contract bullet 16 / AC9) Help-text AC under-tests the contract's own requirement.** Bullet 16 requires the new help line to include "the 'runs on the HOST shell' caveat," mirroring the existing `vibe learn` block's caveat paragraph (vibe:27-30). AC9 only checks "`vibe --help` output contains a `vibe pat` line" — it never asserts the caveat text is present. A lazy Generator adds the one-line usage entry and skips the caveat sentence; AC9 still passes. Add a second assertion for the caveat phrase (or fold it into AC10's presence-check tightening above).

6. **(Contract bullet 15, security) Bearer token is visible in `ps` argv during the 401 probe — not addressed at all.** The contract's "No token values in output" rule covers stdout/stderr and `set -x` traces, but `curl ... -H "Authorization: Bearer $tok" ...` puts the live token into curl's argv, readable via `ps`/`ps aux` on the host for the call's (short) duration to any other local user/process with `ps` access. This is exactly the class of leak the audit brief flagged to check for. It may be an accepted risk (same exposure class as any CLI tool that takes a secret as a flag), but the spec should say so explicitly rather than silently omit it — otherwise a reviewer later flags it as "the spec banned token leaks and this leaks."

7. **(AC1, format) Stream for usage/validation errors is unspecified.** AC1 says "exit 1 with a one-line usage/validation error" but doesn't say stdout vs stderr. Every comparable existing error path in the file (`vibe learn`'s usage block, unknown-flag errors) writes to stderr via `>&2`. Pin it explicitly (stderr) so the Tester and Generator agree without a coin-flip.

8. **(AC7, scope ambiguity) "the test token string" is undefined for AC1.** AC1's cases (`vibe pat a b c`, `vibe pat not-a-slug`) never involve a token at all, so checking "the test token string is absent from captured output" for those runs is vacuously true — harmless, but suggests AC7 was drafted against AC2/AC3/AC5/AC6 and bolted onto AC1 without checking it actually applies. Either scope AC7 to the runs that involve a token (AC2, AC3, AC5, AC6) or state which literal token string is used consistently across all of AC1–AC5 so Tester has one fixture to reuse.

9. **(Contract bullet 9 / no explicit AC) `vibe pat --help` has no carved-out behaviour, unlike every other subcommand in this file.** Under the stated parsing rules, `vibe pat --help` is a 1-arg invocation that fails the "exactly one `/`" slug rule and gets the generic "invalid slug" error — never a help/usage message, unlike `vibe learn` (bare no-arg prints a full usage block, vibe:1519-1527). Minor UX inconsistency, not a hard bug, but worth a decision: either accept it (document the difference) or explicitly special-case `--help`/`-h` before slug validation for consistency with the rest of the CLI's help conventions.

10. **(Files-in-scope / AC9 wording) "the full pre-existing `python3 smoke-test.py` suite passes unmodified" is ambiguous against the Test-location section on the same page.** Test location (spec.md:48-50) mandates the Tester append new `test_task026_*` functions directly into `smoke-test.py` — so the file itself is NOT unmodified by the end of the task. AC9's "passes unmodified" almost certainly means "the pre-existing tests, unchanged, still pass" (a regression gate), but read literally it contradicts the very next section. Reword to something like "all pre-existing test functions (i.e. everything except the new `test_task026_*` additions) still pass, unchanged" to remove the ambiguity.

11. **(Contract bullet 12, minor) Curl exit-code vs `%{http_code}` output isn't reconciled.** The spec says the function returns 0 only for an exact `401` string and 1 for everything else including "curl failure/timeout/000" — this is implementable with `code=$(curl ...) || true; [ "$code" = "401" ]`, ignoring curl's own exit status entirely, which is consistent and fine. Flagging only because the contract doesn't say this explicitly and a Generator who instead gates on curl's exit code (nonzero → treat as automatic non-401) could produce subtly different behaviour on a `403`/`404` that curl still exits 0 for vs a genuine network failure that curl exits nonzero for but might coincidentally still emit some `%{http_code}` text. Low risk given `-w` semantics, but worth one clarifying sentence to avoid two valid-looking implementations that diverge on an edge case (e.g. curl exits 0 but body write fails).
