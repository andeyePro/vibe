## Task summary

`/vsss --sessions X` relies on the `vibe` launcher relaunching `claude --continue "/vsss --resume"` after claude exits. Empirically (3 instances overnight 2026-07-09) interactive claude does NOT exit when the Pro/Max usage window is exhausted – it blocks forever at an interactive usage-limit picker, so the relaunch loop never runs. Research confirms no supported CLI auto-exit/auto-wait/hook exists for this event. Fix: (a) a container-side heartbeat – inline hook entries in the settings.local.json heredoc write an epoch to the project-scoped `/workspace/.vss/heartbeat` around tool activity while a marker exists (project-scoped because the `vibe-claude-config` volume and the `-workspace` transcript slug are SHARED across concurrent vibe instances, so transcript mtime is contaminated); (b) a host-side watchdog – claude is launched backgrounded with its client PID captured, and when the marker is active AND was refreshed during this launcher session AND the heartbeat is provably stale, the watchdog warns, waits a grace period, re-checks, then terminates claude (in-container kill first, host client PID as fallback), dropping the launcher into the existing countdown/relaunch loop – or, on the final window (`remaining=0`), letting the launcher exit cleanly instead of hanging forever. Plus two hardening fixes from PR#12 review: Ctrl-C during the resume countdown deactivates the marker (cancel means cancel – crash-left markers must STILL resume, per Martin), and claude's exit status is captured past the `set -e`-protecting `|| true` and propagated as vibe's final exit code.

Why the three-part kill gate: a crash-left `active=1` marker legitimately persists in a project (crash→resume is wanted behaviour for the AFTER-exit relaunch), but it must never arm a kill against an unrelated interactive session the user opens later in that project. The marker's mtime does the discrimination portably: the `/vsss` skill (re)writes the marker at session start and at the top of every iteration, so during a real `--sessions` run the marker is always newer than the launcher-start reference file, while a crash-left marker predates it. Relaunches inside the auto-resume loop exist only to service the marker, so they arm unconditionally (the freshly-relaunched window may be blocked at the picker before the skill can refresh anything).

## Acceptance criteria

Helper ACs are unit-testable via `VIBE_SOURCE_ONLY=1` sourcing (existing smoke-test.py pattern, cf. `test_vibe_auto_resume_helpers`; function redefinition after sourcing is proven by the `docker()` stub precedent around smoke-test.py:1087). Text ACs assert on the `vibe` script source. All new functions (AC1, AC2, AC3, AC6a, AC6b) live BEFORE the `VIBE_SOURCE_ONLY` guard, adjacent to the existing `auto_resume_*` helpers.

- **AC1 – `auto_resume_deactivate <file>`**: rewrites the marker with `active=0`, preserving all other lines (awk-to-temp + `mv`, matching `auto_resume_decrement`'s portable pattern; temp name `"${file}.tmp.$$"`). Idempotent (running twice leaves `active=0`). Missing file → returns 0 and does NOT create the file. No `.tmp` residue after the call.
- **AC2 – `auto_resume_heartbeat_write <file>`**: writes the current epoch (`date +%s`) as the file's only content, atomically (temp `"${file}.tmp.$$"` + `mv`, no residue). Written value is digits-only and within a few seconds of now. Never returns non-zero (guard the write so `set -e` can't trip; on failure – e.g. unwritable dir – it silently no-ops).
- **AC3 – `auto_resume_stalled <hb_file> <now_epoch> <stall_secs>`**: returns 0 (stalled) iff the file's first line is a digits-only epoch AND `now_epoch - epoch > stall_secs` (strict `>`). Fail-safe returns 1 (NOT stalled) for: missing file, empty file, non-numeric content (e.g. `evil; rm -rf /`), future epochs, and the exact boundary `now_epoch - epoch == stall_secs`. Test cases: gap 1000 vs threshold 900 → stalled (0); gap 100 vs 900 → not (1); gap exactly 900 vs 900 → not (1); missing file → not; garbage content → not; epoch `now+500` → not.
- **AC4 – heartbeat hook entries**: the settings.local.json heredoc in `vibe` gains (i) a `PreToolUse` hook block with matcher `""`, (ii) a `PostToolUse` hook block with matcher `""`, and (iii) one additional command entry for the Stop event (appended to the existing Stop matcher's `hooks` array or as a separate Stop block – either is acceptable), all three with this exact command string (pinned; it runs inside the container as a raw shell string and therefore CANNOT call the host-side AC2 helper – it is a deliberate second implementation of "atomic epoch write"):
  `[ -f /workspace/.vss/auto-resume ] && { date +%s > /workspace/.vss/heartbeat.tmp.$$ && mv /workspace/.vss/heartbeat.tmp.$$ /workspace/.vss/heartbeat; } || true`
  Notes: inside the `'EOF'`-quoted heredoc no expansion happens at write time; `$$` expands in the container-side hook shell at run time. That per-invocation-unique-PID assumption is UNVERIFIED against the hook execution model but benign – the final write is an atomic `mv`, so the worst case is one of two racing epoch writes winning, never a torn file. The command contains no double quotes, so it embeds directly in JSON. The PreToolUse write is what keeps the heartbeat fresh at the START of long-running tool calls (a build/test command can legitimately run for many minutes with no other events).
  Rationale for hooks (vs anything image-baked): the heredoc is rewritten by the launcher on every `vibe` start, so this ships with no Dockerfile rebuild. The new PreToolUse block emits nothing on stdout (no permissionDecision), so it cannot interfere with guard-bash/guard-fs adjudication.
  Structural-equality requirement (NOT byte-identity): parsing the generated JSON, the guard-bash PreToolUse entry, the guard-fs PreToolUse entry, the bell Stop command (`printf '\a' >&2`), the bell Notification entry, `forceLoginMethod: "claudeai"`, and `permissions.defaultMode: "bypassPermissions"` are all still present with unchanged values. Tester verifies by extracting the heredoc body from the `vibe` source (between `cat > "$WORKSPACE/.claude/settings.local.json" << 'EOF'` and the closing `EOF`) and `json.loads`-ing it.
- **AC5 – supervised launch**: `launch_claude`'s body becomes `exec devcontainer exec ...` (it is only ever invoked backgrounded from the supervisor after this change, so the subshell replaces itself and `$!` IS the host-side exec-client PID). NOTE the pre-existing regression test `test_vibe_final_line_no_exec` (smoke-test.py:2672): assertion (a) – no line STARTS with `exec devcontainer exec` – still passes because the function body is indented; assertion (b) – at least one bare non-`exec`-prefixed `devcontainer exec` line exists – is satisfied by AC6a's kill command, which is REQUIRED (see AC6a) to be a literal bare `devcontainer exec ...` line. Do not restructure either in a way that breaks that test.
  Early in the launch path (before the first supervised launch), the launcher creates a session-start reference file: `VIBE_SESSION_REF=$(mktemp "${TMPDIR:-/tmp}/vibe-session-ref.XXXXXX")` (guarded; on mktemp failure the variable stays EMPTY, and per AC6b's gate an empty ref means the watchdog never arms on that window – fail-safe toward NOT killing; the relaunch call sites use the DISTINCT literal sentinel `-`, never empty, so a mktemp failure cannot be confused with unconditional arming).
  A `launch_claude_supervised <resume-args> <initial-prompt> <ref-file>` function: seeds the heartbeat via `auto_resume_heartbeat_write` whenever the marker file exists; launches `launch_claude "$1" "$2" &` (plain `&`, same process group – no `setsid`, no job control); captures `$!`; starts `vibe_stall_watchdog <claude-pid> <marker> <heartbeat> <ref-file>` backgrounded and captures its PID; waits with an explicit per-command guard (`wait "$claude_pid" || rc=$?` – claude legitimately exits non-zero; `set -e` must not trip); stores the status into global `CLAUDE_EXIT`; then reaps the watchdog with individually guarded commands (`kill "$wd_pid" 2>/dev/null || true; wait "$wd_pid" 2>/dev/null || true`). EVERY intermediate command that can fail is individually guarded – the function must be un-trippable under `set -e` and always return 0. Call sites: initial launch passes `"$VIBE_SESSION_REF"` (possibly empty on mktemp failure → never arms); the relaunch inside the auto-resume loop passes the literal `-` (unconditional arming – see gate). No bare `launch_claude ... || true` call remains. `AUTO_RESUME_MARKER` is defined before the first supervised launch.
  NO EXIT-trap changes: the pre-existing clipboard-watcher EXIT trap (vibe:~1629) is not touched. Watchdog lifetime is bounded instead by its own loop condition (claude-PID liveness): if the launcher dies abnormally mid-`wait`, the orphaned watchdog exits by itself within one poll interval of claude's death. Document this bound in the comment block.
  Ctrl-C rationale (document in the comment block, no behaviour change required): while the exec client runs it holds the host TTY in raw mode, so Ctrl-C reaches claude as a byte, not as SIGINT to the launcher – backgrounding does not change this.
- **AC6 – watchdog**, two functions:
  - **AC6a – `vibe_container_kill_claude`**: single responsibility – terminate the claude process inside the project's container via a `devcontainer exec`-based command (e.g. `pkill -TERM -x claude` inside the container), all failure modes swallowed (`|| true` – container may be gone). MUST be written as a literal, bare, non-`exec`-prefixed `devcontainer exec ...` invocation (this is also what keeps the pre-existing `test_vibe_final_line_no_exec` assertion (b) passing – see AC5). Exists as a named function so smoke-test can stub it after sourcing.
  - **AC6b – `vibe_stall_watchdog <claude_pid> <marker_file> <hb_file> <ref_file>`**: loops while `kill -0 <claude_pid>` succeeds; sleeps `${VIBE_STALL_POLL_SECS:-60}` per iteration; the kill gate is THREE conditions, all required: (1) the marker's `active` field is `1` (via `auto_resume_field` – NOT `auto_resume_pending`, so the final window with `remaining=0` is still protected); (2) `<ref_file>` is the literal sentinel `-` (relaunch call site – unconditional arming) OR (`<ref_file>` is non-empty AND exists AND `[ "<marker_file>" -nt "<ref_file>" ]` – portable POSIX `-nt`, no `stat`; the marker was written/refreshed during THIS launcher session, i.e. a `/vsss --sessions` run is genuinely in flight; a crash-left marker predating launch never arms the kill; an EMPTY or missing ref file → gate fails, never arms); (3) `auto_resume_stalled <hb> $(date +%s) ${VIBE_STALL_SECS:-1800}` (default raised to 1800s – with the PreToolUse heartbeat, a stale heartbeat means no hook event for 30 min, and genuine picker-stalls are overnight-scale).
    Why 30 min of no hook events reliably means "not working": (a) hooks fire for SUBAGENT tool calls too – EMPIRICALLY VERIFIED 2026-07-10 in this very container (headless `claude -p` with a PreToolUse probe hook logging `.tool_name + .tool_input.command`; the dispatched subagent's internal `echo subagent-was-here` Bash call produced its own PTU event – see `.vs/cycle-1/spec-critique-iter4.md` Planner resolution note; two older GitHub issues claiming otherwise do not describe the current runtime), so long-running Agent/Task dispatches keep the heartbeat fresh via their internal tool calls – this is also what makes vibe's guard-bash gating of subagent Bash calls work; (b) a single Bash tool call is capped at 10 minutes by Claude Code's max timeout – far under the threshold; (c) background tasks return immediately (PostToolUse fires at dispatch). Residual risk – an exotic tool blocking >30 min with zero session hook events – is accepted: the kill is warned (bell + 120s grace) and non-destructive (the conversation relaunches via `--continue "/vsss --resume"` at the cost of one window decrement). Document the `VIBE_STALL_SECS` override for workloads that genuinely exceed this. On first trigger: prints a loud stderr warning (includes `\a`, the stall threshold, and the cancel instruction "delete .vss/auto-resume"), sleeps `${VIBE_STALL_GRACE_SECS:-120}`, re-checks ALL THREE conditions; only if still met: calls `vibe_container_kill_claude`, sleeps `${VIBE_STALL_KILL_PAUSE_SECS:-10}` (env-overridable, for the in-container kill to land), then kills the host client PID as fallback (`kill "$claude_pid" 2>/dev/null || true`), then returns. No `pkill` against host processes, no pattern-matched kills on the host. All portable constructs (no GNU-only `stat`/`date -d`). With `remaining>=1` the launcher's existing loop then counts down and relaunches; with `remaining=0` the loop is not entered and the launcher exits normally via AC8 – hang fixed on the last window too.
  - Functional tests (enabled by the env overrides + the AC6a stub; all with `VIBE_STALL_POLL_SECS=1 VIBE_STALL_GRACE_SECS=1 VIBE_STALL_SECS=1 VIBE_STALL_KILL_PAUSE_SECS=0`):
    1. POSITIVE kill: stub `vibe_container_kill_claude`; fake claude `sleep 300 &`; marker `active=1`; heartbeat epoch far past; ref file deterministically OLDER than the marker – MANDATED technique: `touch -t 202001010000 "$ref"` (absolute backdate; both BSD and GNU touch accept `-t [[CC]YY]MMDDhhmm`), then write the marker normally → no same-second mtime race. Watchdog (backgrounded) kills the fake claude within a few seconds AND the stub was invoked. Clean up: kill/reap watchdog and fake claude (guarded).
    2. NEGATIVE active=0: marker `active=0`, otherwise same; fake claude is SHORT-LIVED (`sleep 5 &`); watchdog backgrounded → fake claude survives its natural lifetime (still alive at ~2-3s), watchdog exits on its own once the fake claude dies; explicit guarded cleanup of both PIDs regardless.
    3. NEGATIVE stale ref gate: marker `active=1` but the MARKER deterministically older than the ref file (crash-left case) – MANDATED technique: write the marker, `touch -t 202001010000` the MARKER, then create the ref file fresh → same survival assertion as (2).
    4. NEGATIVE empty ref (mktemp-failure fail-safe): marker `active=1`, heartbeat stale, ref argument `""` → same survival assertion as (2) – an empty ref must NEVER arm the kill.
- **AC7 – countdown Ctrl-C trap**: inside the auto-resume loop, an `INT` trap is (re-)armed at the top of EVERY loop iteration, immediately before that iteration's countdown `sleep`; on Ctrl-C it calls `auto_resume_deactivate` on the marker, prints a one-line cancellation notice, and exits 130. The trap is reset (`trap - INT`) after the sleep, before the relaunch, and re-armed again on the next iteration. The trap registers ONLY the INT slot – it must not set, replace, or combine with the EXIT trap (the pre-existing clipboard-watcher EXIT trap must still fire on the exit-130 path; MANUAL-TESTS covers this). Crash paths do NOT deactivate the marker (no EXIT/TERM trap touches the marker).
- **AC8 – exit-status propagation**: the launcher's final statement after the auto-resume loop is an explicit `exit "$CLAUDE_EXIT"` (or exact equivalent) where `CLAUDE_EXIT` holds the LAST claude run's exit status (initial launch's status when the loop never runs). The old `|| true` swallow at the call site is gone (superseded by the supervised function).

- **AC10 – regression gate**: `python3 smoke-test.py` passes with no NEW failures (the 3 pre-existing `[copy]` TTY/clipboard failures are environmental and exempt – they predate this task). Pay particular attention to `test_vibe_final_line_no_exec` (see AC5/AC6a).
