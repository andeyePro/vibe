# Spec Critique — task_016 (cycle 1)

## Concerns

1. **[BLOCKING, AC5/AC6] It is not established that killing the captured PID actually stops the stuck claude process inside the container.**
   `launch_claude` is a bash function whose entire body is a single `devcontainer exec ...` invocation. AC5 says to background `launch_claude` with plain `&` and capture `$!`. When you background a shell function like this, bash forks a subshell to run the function body; `$!` is the PID of *that subshell*, not necessarily the PID of the `devcontainer exec` client process it goes on to run (whether bash's "exec-the-tail-command" optimization collapses the subshell into `devcontainer exec` — making `$!` valid for the real client — is bash-version/context-dependent and not something the spec pins down or requires). Then, even assuming the right host-side PID is captured and killed, `docker exec`/`devcontainer exec` client processes are not guaranteed to forward a `kill <pid>` sent to the *client* into a signal delivered to the *exec'd process inside the container* — that's a different mechanism from a pty-typed Ctrl-C, which the container's tty layer delivers directly. If the in-container claude process is not actually terminated, the watchdog's `kill` just severs the host-side client, `wait` unblocks thinking claude "exited," and the launcher proceeds to relaunch — starting a *second* claude session in the same still-running container while the original stuck one is still alive underneath. That's the exact "concurrent instance / transcript contamination" failure mode the spec itself worries about elsewhere (task_014), self-inflicted by this fix. AC5/AC6 must pin down precisely what process is captured/killed and require some form of verification that the in-container claude process is actually gone (e.g. an explicit `devcontainer exec ... pkill/kill` targeting the in-container PID, or documented reliance on a specific, tested `docker exec` signal-forwarding guarantee) — not just that the host-side `wait` returns.

2. **[BLOCKING, AC5/AC7] Ctrl-C semantics for a normally-running (non-stalled) claude session are undefined and may regress.**
   Today `launch_claude` runs synchronously in the foreground; AC5 replaces that with background-then-`wait`, deliberately keeping the same process group ("plain &, no setsid/job control"). That means once claude is actually running, a user's ordinary Ctrl-C (meant for claude's own UI) lands on the whole foreground process group — the wrapper script and the watchdog subshell included, not just claude. AC7 only specifies an INT trap for the countdown `sleep` between relaunches; it says nothing about what happens if SIGINT arrives while `launch_claude_supervised` is blocked in `wait` on a live, non-stalled claude. Whether the wrapper now silently dies (killing the watchdog and possibly leaving `devcontainer exec`/claude orphaned) or behaves as before is unaddressed and untested. This is exactly the kind of interaction the task brief calls out ("SIGINT reaching background children") and needs its own AC and Tester assertion, not silent inheritance.

3. **[BLOCKING, AC6] The final allowed window (`remaining=0`) has no watchdog protection, reproducing the original bug.**
   `vibe_stall_watchdog` only acts when `auto_resume_pending` is true, and `auto_resume_pending` requires `remaining >= 1`. Marker semantics (per `/vsss --sessions X`) intentionally decrement `remaining` to `0` for the *last* window in the budget — meaning during that final relaunch, `active=1` but `remaining=0`, so `auto_resume_pending` is false and the watchdog takes no action even if that last window also gets stuck at the usage-limit picker. That is precisely the bug this task exists to fix, recurring un-fixed on the very last window of a bounded `--sessions X` run — arguably the most likely window to run out mid-task. The spec should require the watchdog to still detect-and-kill (without relaunching) when `active=1` even if `remaining=0`, so the launcher exits cleanly with a captured exit code instead of hanging forever again.

4. **[MEDIUM, AC5] "The function never returns non-zero" is pinned only at the return-value level, not for intermediate commands under `set -e`.**
   Under `set -e`, any unguarded simple command that fails aborts execution immediately, regardless of what the function eventually intends to `return`. `wait $claude_pid` (claude may legitimately exit nonzero), and `kill`/`wait` on the watchdog PID during reap (which may already be dead, or exit nonzero from its own kill), are all plausible nonzero-returning steps inside `launch_claude_supervised`. As worded, a literal-minded Generator could write code that trips `set -e` mid-function on one of these before ever reaching an explicit `return 0`, aborting the whole script — which is exactly the failure mode AC5 claims to prevent. The AC should explicitly require each such intermediate command be individually guarded (`|| true` / conditional), not just that the function's final return value is 0.

5. **[MEDIUM, AC7] Unclear whether the INT trap must be re-armed on every loop iteration.**
   The auto-resume `while` loop can run multiple iterations when `remaining > 1` (one countdown per relaunch). AC7 says the trap is active "during the countdown sleep" and reset ("trap - INT") "after the sleep." Read literally, a Generator could install the trap once before the loop and never re-arm it after the first `trap - INT`, leaving the second and later countdowns in a multi-relaunch run completely unprotected by Ctrl-C. The AC should say explicitly that the trap is (re-)armed at the top of every loop iteration, immediately before that iteration's `sleep`.

6. **[MEDIUM, AC7] Composition with the existing clipboard-watcher EXIT trap (vibe:1629) is not addressed.**
   The audit brief specifically flags this interaction. Mechanically it should be fine — `INT` and `EXIT` are separate trap slots, and `exit 130` from the new INT trap still fires the pre-existing EXIT trap (clipboard flush + watcher kill) — but the spec never says this explicitly, and nothing requires the Tester to check that the copy-watcher's flush/kill logic still fires on a Ctrl-C-cancelled countdown. A careless implementation (e.g. one that sets `trap '...' INT EXIT` together, or otherwise touches the EXIT trap while adding the INT one) could silently break the clipboard feature on this path with nothing catching it.

7. **[MEDIUM, AC4] "Byte-identical" is ambiguous between literal text-span identity and structural JSON equality.**
   Inserting a new `PostToolUse` key and an additional `Stop` hook command necessarily perturbs adjacent punctuation (trailing commas, brace placement) even when the pre-existing entries are semantically untouched. If the Tester writes a literal byte-substring/diff test, it can be spuriously too strict (fails on harmless reformatting) or too loose (misses a real change, if the substring chosen happens to still be present). The AC should clarify the check is against the *unaffected JSON keys/values* (e.g. parse with `json.load`/`jq` and compare the relevant sub-objects), not a literal byte-range comparison of the whole heredoc.

8. **[MEDIUM, AC4] The literal shell one-liner for the container-side heartbeat write, and the exact "matches all tools" matcher value, are undocumented, unlike AC1-AC3's exact function signatures.**
   AC4 describes intent ("atomic temp + mv", "matcher matching all tools") but doesn't pin the literal matcher string (the existing convention uses `""` for "all") or the substrings a Tester should grep for (e.g. must contain `.tmp`, must contain `mv`, must contain the `[ -f /workspace/.vss/auto-resume ]` guard). Since this hook command runs inside the container as a raw shell string — it cannot call the host-side `auto_resume_heartbeat_write` function from AC2 at all, this is a second, independent implementation of "atomic epoch write" — the spec should say so explicitly and pin concrete substrings, or Tester and Generator may build to different unstated assumptions.

9. **[MEDIUM, AC2/AC12] The heartbeat temp-file naming convention isn't pinned, but AC12 requires gitignoring it.**
   AC1's `auto_resume_decrement` (existing code) uses `"${file}.tmp.$$"`; AC2 should presumably follow the same pattern for `auto_resume_heartbeat_write`, but this isn't stated as a requirement. AC12 then asks for `.vss/heartbeat` "including its temp form" to be gitignored — which needs a wildcard pattern matching whatever suffix convention is actually chosen. If Generator and Tester (and the `.gitignore` entry) don't converge on the same literal suffix, either the residue check or the gitignore pattern can silently miss real files.

10. **[MEDIUM, AC6] Watchdog's poll interval (~60s) and grace period (120s) are hardcoded with no test-time override, unlike `VIBE_STALL_SECS`.**
    A genuine functional/behavioral test of AC6 (not just a structural text-grep) would take minutes (60s+ to detect, 120s grace) — impractical for `smoke-test.py`'s documented "fast, run on every change" convention. The spec should either make these intervals env-overridable the way `VIBE_STALL_SECS` already is, or explicitly bless a stubbing technique for the Tester (e.g. shadow a `sleep` function before sourcing/invoking `vibe_stall_watchdog`, mirroring the existing `docker()` stub pattern in `smoke-test.py`) so cycles don't settle for a weak, structure-only test of the actual watchdog loop.

11. **[LOW, AC3] No explicit boundary-exact test case for `now_epoch - epoch == stall_secs`.**
    The prose pins the semantics (`>`, not `>=`) but the given test cases (1000 vs 900, 100 vs 900) don't exercise the exact boundary. Worth an explicit `now - stall_secs` case to guard against an off-by-one on the comparison operator.

12. **[LOW, AC4] Nesting choice for the new Stop hook command is unconstrained.**
    Whether the heartbeat command is added as a second entry in the *existing* Stop matcher's `hooks` array, or as a wholly separate matcher block, is left open. Either satisfies "byte-identical bell entry" and both are mechanically checkable, so this doesn't block testability — just flagging it as an implementation choice the spec doesn't care to constrain.
