## Concerns

1. **[BLOCKING, AC5/AC10] Adding `exec` to `launch_claude` risks silently breaking the existing regression test `test_vibe_final_line_no_exec` (AC19h), and the spec never notices the dependency.**
   Today there is exactly one occurrence of the string `devcontainer exec` in `vibe` (vibe:1643, inside `launch_claude`). The pre-existing test `test_vibe_final_line_no_exec` requires (a) no line starts with literal `exec devcontainer exec`, and (b) at least one line has `devcontainer exec` (stripped) WITHOUT an `exec ` prefix. Once AC5 turns that line into `exec devcontainer exec ...`, requirement (a) still passes (the line is indented, so it doesn't match the unindented startswith check), but requirement (b) now has *nothing* to satisfy it unless some OTHER new line in the file also contains a bare (non-`exec`-prefixed) `devcontainer exec` invocation. AC6a's `vibe_container_kill_claude` probably supplies that — a `devcontainer exec`-based `pkill` command whose failure needs swallowing with `|| true`, which is incompatible with an `exec` prefix (once `exec` succeeds, the shell is replaced and nothing after it — including `|| true` — ever runs, so a real implementation would almost certainly leave it un-prefixed). But this is an inference, not a spec requirement: nothing in AC5 or AC6a says "AC6a's `devcontainer exec` line must NOT be `exec`-prefixed, because AC19h needs a surviving bare occurrence." A Generator who, say, builds AC6a's kill command via a variable/function indirection rather than a literal `devcontainer exec ...` substring, or who structures AC6a differently, could leave `test_vibe_final_line_no_exec` with zero bare `devcontainer exec` lines to find — an existing regression test (part of AC10's "no NEW failures" gate) fails, without any hint in the spec that this cross-test coupling exists. Fix: state explicitly that AC6a's implementation must contain a bare (non-`exec`-prefixed) `devcontainer exec ...` invocation, and call out the AC19h dependency so the Tester doesn't get blindsided if it fails.

2. **[BLOCKING, AC5/AC6b] The watchdog can kill a genuinely live, working session — not just one stuck at the usage-limit picker — and this is a NEW failure mode the revision itself introduces, not merely inherited risk.**
   Two concrete false-positive paths, both reachable with the spec as written:
   - **Long single tool call.** The heartbeat is only refreshed by `PostToolUse` (fires when a tool call *completes*) and `Stop` (fires when a turn ends). There is no `PreToolUse` heartbeat write. Any single Bash call that legitimately runs longer than `stall_secs + grace` (900s + 120s = 17 minutes by default — a test suite, an `npm install`, a Docker build) produces no heartbeat activity for its whole duration. The watchdog has no way to distinguish "claude is blocked at a picker" from "claude is waiting on a slow tool it correctly launched," and will warn, wait out the grace period, then kill the in-container claude and the host client mid-tool-call. This is exactly the kind of workload `/vsss` autonomous runs are for.
   - **Ordinary interactive use with a leftover marker.** Per Martin's own stated design ("crash-left markers must still resume"), an `active=1` marker from a crashed/finished `/vsss --sessions X` run persists in the project directory indefinitely. If the user later opens a completely ordinary `vibe` session in that same project — not running `/vsss` at all — `launch_claude_supervised` still arms a watchdog because it gates purely on the marker's `active` field, with no way to tell "this launch is the resume attempt for that marker" from "this launch is unrelated interactive work that happens to see a stale marker." `Stop` refreshes the heartbeat at the end of each turn, so if the user takes a normal ~20-minute break to read a long answer, take a call, or think, before typing their next message, the heartbeat goes stale for reasons that have nothing to do with a usage-limit block — and the watchdog kills their live, in-progress conversation out from under them.
   Before this task, a stale marker's only effect was an unwanted *extra relaunch after the user's session had already ended* — annoying but non-destructive. This revision adds a mechanism that can forcibly terminate a session the user is actively using. A stderr warning plus a 120s grace period is not sufficient mitigation for "kill the user's live terminal" — the user may not be watching the terminal at all during a `/vsss` unattended run, and even if they are, 120s is a short window to notice and react. Recommend at least one of: (a) add a `PreToolUse` heartbeat write too, so in-flight tool calls of any length keep the heartbeat fresh and only genuine idle-at-a-picker states go stale; (b) scope watchdog engagement to launches that are themselves servicing the marker (e.g. only arm it on the *relaunch* call sites inside the auto-resume loop, not on the very first `launch_claude_supervised` call of an ordinary `vibe` invocation), so a marker left over from an earlier run doesn't arm a watchdog against unrelated interactive use; (c) raise the default threshold substantially and/or require the grace-period warning to be visible for materially longer before killing. This is the single biggest open risk in the revision.

3. **[MEDIUM, AC6b] The AC6b functional test is mechanically feasible (the `docker()`-stub precedent at smoke-test.py:1087 confirms redefining a sourced function after `source vibe` works), but two of its own timing/lifecycle details are still unpinned, echoing iter-1 #10 only being partially closed:**
   - The "short pause" between the `vibe_container_kill_claude` attempt and the host-PID fallback `kill` is not named, not given a default, and not made env-overridable like `VIBE_STALL_POLL_SECS`/`GRACE_SECS`/`STALL_SECS` are. If the Generator picks something realistic for production (tens of seconds, to give the in-container kill time to actually land before nuking the client), the "kill within a few seconds" functional test becomes exactly as slow as the timings iter-1 #10 was originally trying to fix, and the Tester has no override to work around it.
   - The negative sub-test ("marker `active=0` → fake claude still alive after a comparable wait") has no natural termination: `vibe_stall_watchdog` only returns via its own kill path or when `kill -0 <claude_pid>` fails, so if it's never triggered it will keep polling for as long as the fake claude lives. Run synchronously with `sleep 300` as instructed ("same setup"), this sub-test either blocks for up to 300 seconds or needs the Tester to independently realize it must background the watchdog call and explicitly kill/reap both the watchdog and the fake `sleep` at the end — none of which the spec states. Pin the pause duration (ideally env-overridable) and spell out that the negative case must background the watchdog and explicitly clean up both processes (or just use a short-lived fake claude for that sub-test, e.g. `sleep 5`).

4. **[MEDIUM, AC5] The EXIT-trap watchdog-cleanup mechanism is under-specified and has zero test coverage in either direction.**
   AC5 uses `wd_pid` as the name for the watchdog's captured PID inside `launch_claude_supervised`'s own guarded reap logic, but that reap already runs (kill + wait) before the function returns in the normal case — the EXIT-trap safety net exists specifically for the abnormal case where the script is killed while blocked in `wait "$claude_pid"`, before the function's own reap ever executes. For the pre-existing EXIT trap at vibe:1629 to reach that PID at all, it must be stored in a variable with SCRIPT-LEVEL (global) scope, assigned before the `wait`, under a name both the trap string and the supervised-launch function agree on — none of which AC5 states explicitly. A Generator following ordinary good practice (using `local` inside the function, as the rest of AC5's guarded-reap language implies) would produce a trap edit that silently references an empty/unset variable — the guarded `kill "$X" 2>/dev/null || true` no-ops forever, and the "no orphan watchdog outlives the launcher" claim is false, with nothing catching it: the trap line is never reached by any `VIBE_SOURCE_ONLY=1` test (sourcing returns at the guard well before line 1629), no full end-to-end `vibe` run appears in smoke-test.py, and AC11's manual-test list (stall/kill, remaining=0, Ctrl-C-during-countdown, exit-code propagation, heartbeat refresh) doesn't include "kill vibe itself from outside while claude is running normally and confirm no orphan watchdog remains." Separately, editing this specific trap line at all is exactly the fragile single-quoted, nested-double-quote, `$()`-laden one-liner iter-1 #6 flagged — the revision doesn't pin the literal post-edit trap string the way AC2/AC4 pin their literals, so there's no way for a Tester to check the edit was done correctly even structurally. Recommend: name the global variable explicitly, state it must not be `local`, pin (or require pinning) the edited trap's literal text, and add one manual-test line for the orphan-watchdog case.

5. **[LOW, AC4] The `$$`-in-heredoc PID-uniqueness claim is asserted, not sourced or tested, though the worst case is benign.**
   The spec's claim that `$$` expands per-hook-invocation (fresh PID, no expansion at heredoc-write time) is plausible and consistent with the standard Claude Code hook execution model (each hook command is run as its own subprocess), but nothing in the spec cites that model or tests it, and there's no way to exercise it short of an actual container run. Worth noting for completeness even though it's low-risk: even if two hook invocations somehow shared the same shell (and hence the same `$$`), the destination write still goes through `mv` (atomic rename), so the worst outcome is "one of two near-simultaneous epoch writes wins" — never a torn/corrupt heartbeat file. No action required beyond acknowledging the assumption is unverified.
