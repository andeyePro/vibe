# Spec Critique — task_019: regrettable-content git guard

Audited: `/workspace/.vs/spec.md` against `guard-bash.sh`, `install-claude-extras.sh`,
`project-hygiene.md`, and the `smoke-test.py` test convention.

## Concerns

1. **[BLOCKING] Detection rules / WARN email regex / AC2, AC13 — the guard blocks its own commit convention.** Every commit in this repo (and every vibe-authored commit anywhere, per the mandated trailer) ends `Co-Authored-By: Claude ... <noreply@anthropic.com>`. The WARN-tier email regex `[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}` matches `noreply@anthropic.com` in that trailer, and design bullet 3 + component 2 route the commit message itself through `commit-msg --message`. Component 6's shipped allowlist only claims to cover "hygiene-doc example PII and the maintainer contact link" for *vibe's own repo* — it says nothing about the trailer, and the allowlist mechanism is per-repo/opt-in, so every *other* vibe-managed repo (which all use the same trailer) gets no such allowlist by default. Left as specced, this WARNs (in practice = blocks, see concern 8) on every single commit made anywhere. Fix: either hardcode a scanner-level exemption for the literal `Co-Authored-By: ... <noreply@anthropic.com>` trailer line, or ship a default *global* allowlist entry (not a per-repo opt-in file) covering it, and add an explicit AC asserting a real Claude-authored commit message with the trailer scans clean.

2. **[BLOCKING] Component 7 / AC9 — `vibe audit --history` does not actually scan history.** The component says it scans "every tracked file at HEAD plus all commit messages in history." That is *not* full-history content scanning — it misses a secret that was committed and later deleted (still fully recoverable via `git log -p` / blob access), which is precisely the classic "the flip made it painful" case the task summary cites as the audit's whole reason to exist. As specced, `vibe audit` would report clean on a repo whose history contains a leaked PAT in commit 3 of 10, later removed at commit 4. Fix: scan patches across the full commit range (`git log -p` / `git rev-list --objects --all` + blob content), not just the HEAD tree snapshot.

3. **[BLOCKING] Component 6, 9 / AC1, AC2 — self-referential dogfooding bootstrap gap.** The test fixtures this same spec requires (component 9: `smoke-test.py` cases containing literal fake `ghp_` tokens, `-----BEGIN ... PRIVATE KEY-----` blocks, fake AWS keys, RFC1918 IPs) will themselves be staged and committed as part of landing this task, in *this* repo, which — per the install wiring (component 3) — has the guard active globally in-container by the time the Generator is committing. Nothing in the spec says how the Generator is meant to get those fixtures past pre-commit/commit-msg without either disabling the guard for every dev commit (defeating dogfood value) or the spec explicitly listing the allowlist/exemption path for test-fixture content. This needs a stated bootstrap mechanism (e.g. an explicit `.vibe-content-allow` entry set for `smoke-test.py`'s known fixture strings, or a documented "development commits use `VIBE_CONTENT_GUARD=off`" note) before Generator can build it without repeatedly fighting its own guard.

4. **[BLOCKING] Component 2 / concern (a) — pre-push stdin-to-range mechanics unspecified for the two most common real cases.** Git's pre-push hook receives `<local ref> <local sha1> <remote ref> <remote sha1>` lines on stdin. For the first push of a new branch, `<remote sha1>` is all-zeros — there is no valid `<a>..<b>` range to diff against. For a branch delete, `<local sha1>` is all-zeros — there's nothing to scan. Component 2 just says "`--range` over the refs on stdin" with no resolution rule for either zero-SHA case. First-push-of-a-new-branch is the single most common vibe/`\/vs` push pattern (every cycle branch). If the wrapper naively feeds a zero SHA into `git diff a..b`, it either errors out (blocking every first push) or the Generator special-cases it silently, which could just as easily fail open (skip scanning new-branch pushes entirely, quietly defeating the "backstop" role pre-push is meant to play). Needs an explicit rule in the spec, not left to Generator discretion.

5. **[BLOCKING] AC1–3, AC9 / "Mechanical verifiability" — no AC exercises the actual installed git hooks end-to-end.** ACs 1–3 invoke the scanner directly with `--staged`/`--message`; AC9 invokes `vibe audit`/`--history` directly. None of them prove that the *wrapper scripts* (`pre-commit`, `commit-msg`, `pre-push`) correctly translate git's real hook-invocation contract (argv/stdin) into scanner flags — which is exactly where concern 4's ambiguity lives. All 13 ACs could pass while a real `git commit`/`git push` through the installed hooks in a throwaway repo silently no-ops or crashes. Add at least one integration-level AC: install the hooks, `git init` a throwaway repo + local bare remote, do a real first commit and a real first push of a new branch, and assert the guard actually fires/passes as expected.

6. **[MEDIUM] Design bullet 5 / AC5 — WARN's "explicit confirmation" is undefined for the dominant execution context.** Commits in this environment are made by Claude Code via the Bash tool under `bypassPermissions`, with no attached TTY. There's no described mechanism for an interactive "confirm to proceed" on a WARN — the only paths given are `VIBE_CONTENT_GUARD=off`/`VIBE_ALLOW_COMMIT=1` (an env override) or `git commit --no-verify`. Both are functionally identical to how a BLOCK is bypassed. So in the actual primary use case, "two-tier: BLOCK vs WARN" collapses into one behaviour (both need the same override) — the distinction only shows up in exit-code-adjacent text output, not in what the operator (agent) has to do. Not necessarily wrong, but the spec should say so explicitly rather than imply WARN is a softer gate in practice.

7. **[MEDIUM] Concern (d) — triple-scan re-litigates already-approved content on every push.** `pre-commit`, `commit-msg`, and `pre-push` each independently scan overlapping content (staged diff, message, then push-range diff+messages). An override given at commit time (per-invocation, "never silent," not persisted anywhere) does not carry forward — the same already-committed, already-approved WARN will re-fire at `pre-push` for every push until the branch merges, and again on every subsequent push if more commits are added ahead of it. This is exactly the "cries wolf → gets disabled" failure the invariants section names as make-or-break. Spec should pick a stance: either persist per-invocation overrides against the specific commit SHA (more machinery) or explicitly accept the repeat-WARN friction and say so.

8. **[MEDIUM] Concern (f) — allowlist matching semantics unspecified.** Component 6 says entries are "one regex or literal per line" matched against "the matched text" of a finding, but doesn't specify: anchored or substring match, regex-metacharacter escaping for what looks like a literal (e.g. an IP address's dots are regex "any char"), case sensitivity, or whether "matched text" is the whole flagged line vs just the captured token (matters a lot for the BLOCK secret-assignment rule, where the "match" spans a `key=value` pair). AC6 tests this only loosely ("verified against vibe's own repo content"). Pin the contract precisely so Generator and Tester don't independently guess different semantics.

9. **[MEDIUM] Component 4 — two documented opt-out syntaxes, only one has any AC.** The per-repo opt-out is specified as *either* a `.vibe-content-guard: off` line in a `.vibe/config`-style marker *or* a `.vibe-content-guard-off` marker file. AC7 only exercises the marker-file form. The `.vibe/config` line form (including its precedence if both files exist, or if the line reads `on`) has zero mechanical coverage and is pure surface area for a bug. Recommend dropping to a single syntax, or adding a matching AC.

10. **[MEDIUM] AC8 — permissive test language invites a weak assertion.** "assert via a stubbed `git config` capture **or** the emitted command" lets a passing test prove only that the install script *calls* `git config --global core.hooksPath ...` with the right arguments, not that the resulting git config state actually resolves that way (the existing `smoke-test.py` convention elsewhere uses real temp-`HOME` + real subprocess invocation rather than binary stubbing — e.g. `test_help`). Tighten to: run the real install script against an isolated `HOME`/`GIT_CONFIG_GLOBAL`, then assert `git config --global core.hooksPath` actually returns the expected path.

11. **[MEDIUM] AC6 — coupling a mechanism test to live doc content risks drift and false confidence.** AC6 partly verifies the generic allowlist-suppression mechanism and partly asserts a fact about the *current* text of `project-hygiene.md` ("its example-PII hygiene lines do not trip the guard once the allowlist ships"). If that doc gains a new PII-shaped example later without a matching allowlist entry, nothing catches it — the mechanism test still passes because it's exercising old, already-covered strings. Split into (a) a synthetic fixture proving the mechanism works in isolation, and (b) if the real-doc check is kept, make it re-derive its expected findings from the doc's current content rather than a hardcoded example, so drift is caught rather than papered over.

12. **[LOW] Output format for BLOCK vs WARN not pinned.** Component 1 says the scanner "prints file/line/class/rule" but gives no exact line format. AC3 needs output to be "distinct... in output" — without a pinned format (e.g. a literal `BLOCK\t`/`WARN\t` prefix token), a test could pass by accident (e.g. grepping for the substring "WARN" that happens to also appear inside a BLOCK rule's name or description) without genuinely verifying the classifications are distinguishable. Recommend the spec pin a minimal machine-parseable line shape.

13. **[LOW] Exit code 2 reused across two unrelated protocols.** guard-bash.sh's exit 2 is a specific signal in the Claude Code PreToolUse hook protocol ("block this tool call"). The new git hooks are real git hooks, where git only cares about zero vs non-zero — there's no functional collision, but reusing "2" invites a future maintainer to assume it carries the same meaning here. Cosmetic; a comment or a different code (e.g. 1) would remove the ambiguity for free.

14. **[LOW] `core.hooksPath` is global — interaction with `/repos/*` shared checkouts untested.** Because component 3 sets `core.hooksPath` at `--global` scope, every rw shared repo mounted at `/repos/<name>` also gets scanned on commit, with no AC or out-of-scope note addressing this interaction with task_017's per-repo credential routing. Likely fine, but currently just unstated, not designed-for.

15. **[LOW] AC10 mostly duplicates existing coverage.** `test_task009_guard_bash_gitpush_and_block_beats_ask` (already in `smoke-test.py`) already exercises the force-push-no-lease block. AC10 as a fresh, separately-run assertion adds little beyond what AC12 ("full pre-existing suite still green") already guarantees — harmless to keep, but it's not new coverage, just emphasis.

16. **[LOW] Brain2 framing inconsistency.** Component 4 frames the opt-out marker as "how a brain2/private repo is exempted," but the user's existing standing instruction (CLAUDE.md) already forbids running git directly against `/brain2` at all — the Mac auto-commits it. The opt-out's stated flagship use case may never actually be exercised by an in-container `git commit`. Not a functional bug, just imprecise motivating language that could mislead a future reader into thinking brain2 needs this marker.

## Verdict

**revise**

The spec's mechanics are broadly sound (two-tier, allowlist, opt-out, override are the right primitives) but concerns 1–5 are not edge cases — they hit the guard's two headline claims (won't break ordinary commits; will catch history secrets before a Private→Public flip) and the very commits that will land this task. Concern 1 in particular (the `Co-Authored-By: ... <noreply@anthropic.com>` trailer) would make the guard fire on every commit in every vibe-managed repo from day one, which is the exact "cries wolf → gets disabled" failure the invariants section warns against. These need spec-level resolution before Generator starts, not discovery mid-cycle.
