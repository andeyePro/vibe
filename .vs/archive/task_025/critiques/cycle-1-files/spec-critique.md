# Spec Critique — task_025 (cycle 1)

Verdict: **revise**

## Concerns

1. **BLOCKING (AC2) — shebang predicate false-positives on non-shell interpreters.** The pinned rule is "first line starts `#!` and contains `sh`." Empirically checked: `"sh" in "/usr/bin/env fish"` is `True` — a `#!/usr/bin/env fish` file would be wrongly classified as a shell script, because "fish" contains "sh" as a substring. (Correctly, `#!/usr/bin/python` does NOT contain "sh," so that one's fine — but "fish" is a real false positive.) `csh`/`tcsh`/`zsh` also contain "sh" but those genuinely are shells, so a naive fix to "ends with sh" isn't quite right either (need `#!/bin/csh` to still match if csh is meant to count — spec doesn't say). This needs to be pinned precisely before code is written (e.g., basename of the interpreter path matches a whitelist, or a word-boundary regex), or Generator and Tester will independently guess different fixes and diverge on exactly the adversarial case (a "fish"-style near-miss) the task brief asked to check for.

2. **BLOCKING (AC2 vs. the env-seam contract) — the shebang predicate has no way to be unit-tested as specified.** AC2 says "test the shebang predicate directly against crafted files." But the env seam (AC3/AC4) is checked FIRST in `scripts()` and, when set, completely bypasses the default-set/git-hooks-scanning branch where shebang detection lives — so `CODE_CHECK_SCRIPTS` cannot be used to exercise the predicate at all. The pinned contract never names a standalone, importable function for the predicate (e.g. a module-level `_is_shell_shebang(path) -> bool`). Without one, the only ways left for the Tester to satisfy AC2 are (a) resurrect the exact "copy the source, monkeypatch REPO, patch in synthetic files" technique this task exists to retire, or (b) drop crafted fixture files directly into the real `devcontainer/git-hooks/` tree during a test run — both bad. Pin a named, directly-callable predicate function in the contract so Generator commits it and Tester can import and call it standalone.

3. **BLOCKING (default-set assembly) — ambiguous whether `.sh` files in `git-hooks/` are included unconditionally-by-glob or also gated by the shebang check.** The spec's two adjacent sentences support two different algorithms: (a) `sorted(glob("*.sh"))` unconditional (matching the existing `devcontainer/*.sh` and `hooks/*.sh` patterns) concatenated with shebang-filtered extensionless files, vs. (b) shebang-detection applied uniformly to every file in the directory including `.sh` ones (a single flat `sorted(iterdir())` pass). Both satisfy today's fixture (all 4 files are well-formed bash) so no current AC catches the divergence, but a Generator and a Tester reading independently could each build a self-consistent implementation that disagrees on: a hypothetical shebang-less `.sh` file (included under (a), excluded under (b)), and on sort order (two sorted sub-lists concatenated vs. one fully-interleaved sorted list — e.g. does `vibe-content-scan.sh` always sort last, or interleave alphabetically with `commit-msg`/`pre-commit`/`pre-push`?). Pin one specific algorithm.

4. **BLOCKING (test isolation, protects AC8 + two pre-existing frozen tests) — `CODE_CHECK_SCRIPTS` env-var leakage risk across the fixed test sequence.** AC1 explicitly permits testing `scripts()` "in-process," which (since this is a flat script, not pytest with fixtures) means mutating the real `os.environ["CODE_CHECK_SCRIPTS"]` directly — a pattern this file otherwise never uses (every one of the ~100 existing `env = {**os.environ, ...}` sites in `smoke-test.py` passes a fresh dict into `subprocess.run(env=...)`, never mutates the parent's environ). `main()` calls tests in a fixed, hardcoded sequence; the new git-hooks/env-seam tests will land in the `test_code_check_json_*` block (lines 10776–10785), and TWO already-frozen regression tests run later in that same sequence and would inherit any leaked value: `test_learning_code_check_clean` (def line 2069, called at 10862) and `test_task009_code_check_clean` (def line 4119, called at 10926) — both do a bare `run(["python3", str(CODE_CHECK)], cwd=REPO)`, and `run()` defaults to `env=os.environ.copy()` when no `env=` is passed (smoke-test.py:73-81). Given the pinned "no `.exists()` filter on override" rule, a leaked stale/deleted temp path would make these break loudly and misleadingly, unrelated to any real regression. Spec should explicitly require: any env-var mutation for this feature is either subprocess-scoped only (never touches the parent process's `os.environ`) or is wrapped in try/finally to restore state immediately.

5. **Minor (traceability) — spec mislabels the three existing `_patched_code_check` call sites.** Spec says "the clean-exit / findings-schema / multi-file JSON-count tests." Grep-verified actual call sites: `test_code_check_json_finding_schema`, `test_code_check_json_findings_exit1_and_count`, `test_code_check_json_summary_counts`. None is the "clean-exit" test — `test_code_check_json_clean_exit_and_valid_json` does NOT call `_patched_code_check` (confirmed by reading its body: it runs the real `CODE_CHECK` directly, no fixture). Fix the naming so the Tester doesn't go hunting for a nonexistent fourth call site.

6. **Minor (AC7 mechanical weakness) — "brittle string is gone" grep is under-specified and, as literally described, self-fulfilling.** A test that just asserts "the exact old `scripts()`-body literal is absent from `_patched_code_check`" is trivially true the instant the reshape lands and guards against nothing recurring later. Pin something structural instead, e.g. "no `.replace(` call in `_patched_code_check` whose argument text contains the substring `def scripts()`," so the regression guard protects against the *pattern* recurring, not just the literal string never coming back verbatim.

7. **Minor (robustness) — shebang-line reading has no defined behaviour for empty/non-UTF-8 files.** Doesn't bite today (all four current files are clean UTF-8 bash with a real first line), but the predicate is written as a general rule for the directory, and a naive `Path.read_text().splitlines()[0]` read would raise `UnicodeDecodeError` on a binary file or `IndexError`/return `""` on an empty one — crashing `scripts()` (and thus every `code-check.py` invocation) rather than just excluding the file. Pin defensive reading (catch decode errors, treat unreadable/empty as "not shell").

## Not a concern (verified, no action needed)

- The "freeze-anchor exception" invoked to justify amending `_patched_code_check` is well-grounded, not an overreach: `_patched_code_check` (smoke-test.py:451) sits well before the nearest explicit "frozen" section marker in the file (line 7776, task_017 C3), and there's direct repo precedent for retiring a freeze-anchored test that blocked legitimate future work (`AUTONOMOUS-MEMO-2026-05-02.md`: "Retired one stale freeze-anchored test... following the auto-memory rule").
- AC5's factual premise checked empirically: all four `devcontainer/git-hooks/` files (`commit-msg`, `pre-commit`, `pre-push`, `vibe-content-scan.sh`) are shellcheck-clean today (`shellcheck --shell=bash --severity=warning`, exit 0 on all four).
- Only three call sites of `_patched_code_check` exist (grep-confirmed), matching the spec's count claim — just mislabeled per concern 5.
- `git-hooks/` has no subdirectories, consistent with the out-of-scope "no recursion" note.
