# Spec Critique — task_025 (cycle 1, iteration 2)

Verdict: **revise**

## Resolution check (iter-1 concerns)

1. **RESOLVED — shebang predicate whitelist.** Contract now: interpreter = basename of first token, UNLESS first token's basename is `env` (then basename of second token); whitelist `{sh, bash, dash, ksh, zsh, ash}`, exact case-sensitive match. Traced by hand: `#!/usr/bin/env bash` → tokens `["/usr/bin/env","bash"]` → first-token basename `env` → use second token `bash` → in whitelist → `True` ✓. `#!/bin/sh` → tokens `["/bin/sh"]` → basename `sh` → `True` ✓. `#!/usr/bin/env fish` → second token `fish` → not in whitelist → `False` ✓ (the substring false-positive is gone). `#!/usr/bin/python3` → basename `python3` → `False` ✓. Algorithm is unambiguous and matches all 8 AC2 cases.
2. **RESOLVED — named importable predicate.** `is_shell_script(path: Path) -> bool` is pinned as a module-level function, independent of the env seam, so AC2 can import and call it directly without touching `CODE_CHECK_SCRIPTS` or the real `git-hooks/` tree.
3. **RESOLVED — one flat algorithm, no separate `*.sh` glob.** Spec now pins a single `sorted(p for p in devcontainer/git-hooks/ ... if p.is_file() and is_shell_script(p))` pass, explicitly "NOT a separate `*.sh` glob," with sort order and result (`commit-msg`, `pre-commit`, `pre-push`, `vibe-content-scan.sh`) spelled out. No room left for the two-algorithm divergence flagged in iter1.
4. **RESOLVED — env-leak isolation required.** AC3 now explicitly mandates subprocess-scoped mutation or try/finally restore, and names the two frozen tests (`test_learning_code_check_clean`, `test_task009_code_check_clean`) at risk of inheriting a leaked override.
5. **RESOLVED — call-site names corrected.** Files-in-scope and AC6 both now use the grep-verified names and explicitly note the clean-exit test does not use the fixture.
6. **RESOLVED — AC7 structural pattern-grep.** AC7 now targets "no `.replace(` call whose argument text includes the substring `def scripts()`" — a pattern check, not a literal-string self-fulfilling check.
7. **RESOLVED (for the cases it enumerates) — defensive first-line read.** Contract requires `open(..., 'r', errors='ignore')`, catches OSError/UnicodeError, and treats an "empty/first-lineless file" as `False`, never raising. This closes the crash mode iter1 flagged for binary/empty files. See new finding below for a related gap this clause does not cover.

All four blocking and three minor iter-1 concerns are resolved as written. No regressions introduced by the resolutions themselves.

## New adversarial pass (fresh, on the revised text only)

1. **BLOCKING (new) — the interpreter-resolution algorithm has unguarded short-token-list paths that violate the predicate's own "never raise" promise, and at least one of the triggering forms is real-world, not pathological.** The pinned algorithm reads: "the interpreter is the last path component of the first token, UNLESS that first token's basename is `env`, in which case it is the basename of the *second* token." This assumes a token always exists at the position being read. Three concrete inputs break that assumption:
   - `#!/usr/bin/env` (env invoked with no interpreter argument at all) → tokens = `["/usr/bin/env"]`. First-token basename is `env`, so the rule says "use the basename of the second token" — there is no second token. A literal implementation (`tokens[1]`) raises `IndexError`.
   - `#!/usr/bin/env -S bash -x` (the GNU/BSD `env -S` split-string form, explicitly asked about in this review, and a real, documented, currently-used idiom — not invented) → tokens = `["/usr/bin/env","-S","bash","-x"]`. First-token basename `env` → the rule points at the *second* token, which is `-S`, not `bash`. Basename of `-S` is `-S`, not in the whitelist → the predicate returns `False` for a file that genuinely is a bash script. This is a silent misclassification, not a crash, but it's wrong, and the spec explicitly asked whether `-S` is handled — it is not.
   - `#!` alone, nothing after (asked about explicitly) → after stripping `#!`, the remainder is `""`, which `.split()` reduces to `[]` — zero tokens. "The interpreter is the last path component of the first token" has no first token to read. A literal implementation (`tokens[0]`) raises `IndexError`. This case is distinct from the "empty/first-lineless file" guard already in the contract (that guard covers a file with no first line or an empty first line — `"#!"` is a non-empty first line, so it does not fall under that clause as written).

   All three are IndexError-or-silent-wrong-answer paths through code that the contract promises "never raise[s]." Because `is_shell_script` is called from inside `scripts()`'s default-set assembly, an uncaught exception here does not just misclassify one file — it crashes `scripts()`, and therefore every bare invocation of `python3 code-check.py` (including the two frozen clean-exit regression tests), the moment any file with one of these three shebang shapes lands in `devcontainer/git-hooks/`. None of the 8 AC2 test cases exercises any of the three, so Generator and Tester can both pass every stated AC while shipping this gap — it won't be caught by the acceptance criteria as currently written.

   Recommend pinning explicit guards before Generator starts: (a) if the split produces zero tokens, return `False`; (b) if the first token's basename is `env`, walk forward past any subsequent tokens that start with `-` (flags, e.g. `-S`, `-i`, `-u NAME=VAL`) to find the first non-flag token as the interpreter, and if none exists, return `False`. This also fixes the `-S` misclassification, not just the crash paths.

## Minor / trivia (new)

- **`open(errors='ignore')` + one-line read on a truly binary file with no newline byte anywhere.** `readline()` (or equivalent) would read the entire file into memory as one "line" before returning, since there's no `\n` to stop at. Not a correctness bug (decoding with `errors='ignore'` cannot raise, and the result is checked against `#!` same as any other string), but it's an unbounded-read performance smell for a predicate documented as reading "the file's first line." Worth a `f.readline()`-with-a-length-cap or explicit note that this is accepted as out-of-scope for a lint-only file-selector, but not blocking — no realistic file in `git-hooks/` today or foreseeably triggers it.
- **Trailing-slash interpreter path, e.g. `#!/bin/sh/`.** `os.path.basename("/bin/sh/")` returns `""` (empty string), which correctly falls outside the whitelist, so the predicate returns `False` for a shebang that's arguably intended as `sh`. This is almost certainly fine to leave as `False` — a trailing slash on an interpreter path is not a form any real toolchain emits — but flagging since the review explicitly asked about it. No action needed; behavior is well-defined and safely conservative (excludes rather than crashes or false-includes).
