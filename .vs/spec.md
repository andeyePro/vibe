# Spec — task_017: shared-repos — mount a private repo into multiple projects' vibe containers (FOSS-core freemium dev)

Design settled with Martin 2026-07-10/11 (conversation record in `.vss/sessions/2026-07-11T03-13-53Z.md`). The seven decided slices are NOT open for re-litigation; this spec pins their mechanics. Delivery is 4 commits/cycles, dependency-ordered. Generator tier: sonnet for C1-C3, opus for C4 (credential routing). The repo's security-review agent reviews C4 (mandatory) and C1 (advisory) before commit.

## Task summary

Multiple public `<x>andeye` apps need one private repo (e.g. `andeyePro/andeyePro`) mounted into their vibe containers for live cross-repo development, permanently configured per-project, with conflict handling and honest blast-radius accounting. A project commits a declaration of WHAT it uses (repo slug only); each machine registers WHERE that repo lives locally (and its PAT). Shared repos mount read-only at `/repos/<name>` by default; write access is a per-project intent refereed by a per-session single-writer lock in the shared checkout. Community contributors without the private repo see nothing. Tokens never widen: each repo — project and shared alike — is served by its own single-repo PAT via `credential.useHttpPath`.

## Pinned names and formats (all cycles)

- **Declaration file**: `/workspace/.vibe-repos` — COMMITTED (must NOT be caught by the vibe-managed gitignore block; add a smoke check proving `ensure_project_gitignore`'s block does not match it). One entry per line: `<owner>/<repo> <ro|rw>`; mode optional, default `ro`. Slug charset `[A-Za-z0-9._-]+/[A-Za-z0-9._-]+` — reject anything else at parse time (helper `shared_repos_parse <file>`, echoes validated `slug mode` lines only; malformed lines are skipped with a stderr warning naming the line). Comments (`#`) and blank lines allowed.
- **Machine registry**: `~/.vibe/repos` (chmod 600 like tokens). Lines `slug=path` — slug validated as above; path is everything after the first `=`, rejected if it contains a newline or double-quote; must be an absolute path. Helper `repos_registry_lookup <slug>` echoes the path or nothing (mirrors `lookup_token` / `auto_resume_field` fail-safe discipline).
- **Mount point**: `/repos/<repo-name>` (basename of slug). Collisions between two slugs with the same basename are rejected at `vibe repos add` time. Repo basenames beginning with `.` are RESERVED (the `.signals` sidecar namespace lives beside the repo mounts): rejected at `vibe repos add`, and skipped-with-warning at mount assembly if one appears in a hand-edited declaration. In-container scanners use the `${VIBE_REPOS_DIR:-/repos}/*/` glob, which never matches dot-dirs — keep it that way.
- **Signals sidecar**: `<shared-checkout>/.vibe-signals/` — ONE host-side directory per shared checkout holding all coordination state; created host-side with `mkdir -p` by BOTH `vibe repos add` (at registration) and the mount-assembly step (defensively, before any bind referencing it) so it always exists with the invoking user's ownership BEFORE Docker sees it — never rely on Docker's bind auto-vivification (engine-dependent, can create it root-owned and break the non-root container user). Mounted rw at `/repos/.signals/<name>/` even when the code mount is ro (this is what lets an ro session file a claim).
- **Lock**: `<shared-checkout>/.vibe-signals/rw-lock.d/` — a DIRECTORY, because `mkdir` is the atomic acquire (no test-then-write race); metadata file `meta` inside with KEY=VALUE lines `project=<name>`, `pid=<launcher pid>`, `since=<epoch>` (pid/since digits-only on read). Stale-reclaim: if `kill -0 <pid>` fails, `rm -rf` the lock dir and retry the `mkdir` ONCE.
- **Claim request**: `<shared-checkout>/.vibe-signals/rw-request` — KEY=VALUE `project=`, `since=` (same parse discipline). Overwritten by the newest claimant (last-claim-wins is fine; the holder sees a name either way).
- **Runtime manifest** (container-visible): `/workspace/.vibe/shared-repos.manifest` — written by the launcher at each launch, one line per mounted repo: `name mode slug` — lets the in-container fragment/statusLine/commands know what's mounted without guessing. `.vibe/` is already gitignored by the managed block.
- **Gitignore additions**: `.vibe-signals/` — two owners, both required: (a) `ensure_project_gitignore`'s managed block gains the entry (covers every vibe project, including a shared repo that is itself a vibe project); (b) `vibe repos add` ALSO directly ensures `.vibe-signals/` is gitignored in the SHARED checkout at registration time (covers shared checkouts that are not vibe projects and have never launched vibe). NOT `.vibe-repos` (committed by design).
- **Per-slug token env**: `VIBE_SHARED_TOKEN_<SLUG_SANITISED>` where sanitisation = uppercase, every non-alphanumeric → `_` (e.g. `andeyePro/andeyePro` → `ANDEYEPRO_ANDEYEPRO`). Collision between two configured slugs after sanitisation → reject at `vibe repos add` time. Tokens stored in `~/.vibe/tokens` under the slug key (the existing per-repo PAT store; `lookup_token` already handles arbitrary keys). Passed to the container via **remoteEnv ONLY** (mirroring GITHUB_TOKEN / ZOTERO_API_KEY — never containerEnv, never the firewall/postStart layer).
- **Env knob for tests**: the statusLine command and any in-container helper that scans `/repos` must read the root from `${VIBE_REPOS_DIR:-/repos}` so smoke fixtures can point it at a temp dir.

## Cycle 1 — registry + declaration + mounts + header (slices 1a, 4)

- **AC1 – `vibe repos` subcommand** (host-side; dispatch mechanism pinned: a literal `if [ "${1:-}" = "repos" ]` block placed BEFORE `parse_vibe_args "$@"`, exactly like the `learn` block at vibe:1401-1405 — otherwise `vibe repos add ...` falls through to parse_vibe_args's positional catch-all and is treated as a project name): `vibe repos add <owner/repo> [path]` validates the slug, resolves/validates the path (default: prompt; must contain a `.git`), writes the registry entry, calls `shared_repo_ensure_signals <checkout>` — a NAMED helper (before the VIBE_SOURCE_ONLY guard, unit-testable against a temp dir) that `mkdir -p`s the `.vibe-signals/` sidecar AND ensures `.vibe-signals/` is gitignored in the shared checkout, both guarded/idempotent — appends the declaration line to the CURRENT project's `.vibe-repos` if absent, and — if no PAT exists under the slug — prints the fine-grained-PAT mint instructions (same flow as the project PAT, repo pre-selected) and stages it in `~/.vibe/tokens`. `vibe repos remove <owner/repo>` removes the declaration line from the current project (registry entry retained — other projects may use it; `--purge` also drops registry + prompts about the token). `vibe repos list` shows declared repos for the current project with resolution status (`mounted ro/rw` / `not configured on this machine` / `BROKEN: <reason>`). All three exit non-zero on validation failure with a one-line reason. Both `add` and `remove` end by noting a relaunch is needed (mounts are fixed at container creation).
- **AC2 – parse/lookup helpers** before the `VIBE_SOURCE_ONLY` guard, unit-testable: `shared_repos_parse`, `repos_registry_lookup`, `shared_repo_env_name <slug>` (sanitiser), each with the fail-safe/charset behaviour pinned above (tests: valid line, bad slug skipped with warning, comment/blank skipped, missing files echo nothing, injection strings like `foo/bar;rm -rf /` rejected, env-name sanitisation cases, and dot-prefixed repo basenames rejected/skipped per the reserved-namespace rule).
- **AC3 – mount assembly**: `_build_override_config` gains shared-repo bind triples — for each declared slug that resolves via the registry to an existing directory: bind at `/repos/<name>`, ro flag from the declaration (C1 may hard-pin ro; rw arrives in C2). Unresolvable-and-unregistered → silently skipped. Registered-but-broken (registry entry exists but path missing/not a git repo, or declared token key absent) → still skipped BUT recorded for the header warning. Mount assembly also `mkdir -p`s each resolved checkout's `.vibe-signals/` sidecar and adds its rw bind (see Pinned names); that mkdir is GUARDED — on failure (read-only or network volume) the repo is demoted to registered-but-broken with reason `sidecar unwritable`, BOTH its binds are skipped, and assembly continues with sibling repos (never a `set -e` abort) — Cycle-1 Testers must NOT assert one-mount-per-repo exclusivity; each shared repo contributes TWO bind triples (code + sidecar) from C1 onward.
- **AC4 – launch header**: for each MOUNTED shared repo, one line `◆ shared repo: <slug> at /repos/<name> (<ro|rw>)`. For each registered-but-broken repo, a loud warning line naming the reason. For never-registered declarations: NOTHING is printed (community-contributor silence — test asserts the code path distinguishes "no registry entry" from "registry entry broken").
- **AC5 – runtime manifest** written per launch (or truncated to empty when no shared repos) immediately after mount assembly and STRICTLY BEFORE `devcontainer up` (vibe ~1769) — postStart's install-claude-extras.sh consumes it in the SAME launch for AC6's safe.directory entries; writing it any later hands postStart a one-launch-stale manifest, which would break exactly the critical path (the relaunch right after `vibe repos add`).
- **AC6 – git ergonomics in-container**: there is NO existing safe.directory mechanism in this codebase to mirror (verified — /workspace has never needed one, its ownership maps cleanly on the primary Mac platform). Specify directly: install-claude-extras.sh (runs every container start) reads the runtime manifest and runs `git config --global --add safe.directory /repos/<name>` for each mounted shared repo (literal per-repo entries; no `*` wildcards). Idempotency: de-duplicate before adding. Plus `GIT_OPTIONAL_LOCKS=0` guidance for ro checkouts (pin the exact mechanism: remoteEnv var or the fragment's instruction — Generator's choice, documented).
- **AC7 – gates**: code-check clean; smoke-test additions for AC1-AC5 (explicitly including: `ensure_project_gitignore`'s managed block contains `.vibe-signals/`; `shared_repo_ensure_signals` creates sidecar + gitignore entry idempotently in a temp checkout and fails soft on an unwritable one) (subcommands exercised via `VIBE_SOURCE_ONLY` function calls and/or the smoke harness's existing subcommand-test pattern); README section + CHANGELOG entry in the same commit.

## Cycle 2 — rw intent + single-writer lock (slice 2)

- **AC8 – intent**: `.vibe-repos` mode field `rw` is honoured: launcher attempts lock acquisition at launch; on success the bind is rw and the header says so; on failure (lock held by a live pid) the bind FALLS BACK to ro and the header loudly names the holder project + since-time. `vibe repos add --rw <slug>` sets/updates the mode in the declaration.
- **AC9 – lock helpers** (before the guard, unit-testable): `shared_repo_lock_acquire <checkout> <project> <pid>` (atomic `mkdir`; writes meta; stale-reclaim exactly once when holder pid dead; returns 0 acquired / 1 held with holder details echoed), `shared_repo_lock_release <checkout> <project> <pid>` (releases ONLY if meta matches both project and pid — never someone else's lock), `shared_repo_lock_holder <checkout>` (echoes `project pid since` with digits-only discipline). Functional tests: acquire/contend/stale-reclaim/wrong-owner-release-refused, all with real processes (`sleep` PIDs) per the task_016 watchdog-test pattern.
- **AC10 – exit release without clobbering existing traps**: vibe's EXIT handling is consolidated into a single `vibe_on_exit` dispatcher installed once, with an append-mechanism (e.g. `VIBE_EXIT_HOOKS+=('...')` array) — the existing Darwin clipboard-flush trap body migrates into it UNCHANGED (its current behaviour is pinned by existing smoke checks and MANUAL-TESTS; the migration must keep those green), and lock release registers alongside. The dispatcher MUST invoke every registered hook individually guarded (`eval "$hook" || true` or equivalent) — the script runs under `set -euo pipefail`, and an unguarded refusing hook (e.g. `shared_repo_lock_release` in the routine never-held-the-lock case) would abort the trap handler and silently kill every later-registered hook, exactly the clipboard-flush regression task_016 refused to risk. No `trap ... EXIT` remains outside the dispatcher installer. Crash paths: a killed launcher leaves the lock; the NEXT acquirer's stale-reclaim handles it (pid dead) — no other cleanup mechanism.
- **AC11 – gates** as AC7 (smoke for AC8-AC10; MANUAL-TESTS gains lock scenarios: contended launch falls back ro with named holder; stale lock reclaimed after kill -9; clipboard flush still works on exit — regression guard for the trap migration; plus a functional smoke test proving a REFUSING earlier-registered hook does not suppress a later-registered hook in the dispatcher).

## Cycle 3 — claim/handoff + /repo command + CLAUDE.md fragment (slices 3, 6)

- **AC12 – `/repo` in-session command** (`devcontainer/commands/repo.md`, synced by install-claude-extras like /vs et al): `/repo add <slug> [--rw]` edits `/workspace/.vibe-repos` (validated) and tells the user the machine-side step (`vibe repos add <slug> [path]` on the Mac) plus the relaunch note — it must be explicit that the container CANNOT touch `~/.vibe` on the host; `/repo remove` edits the declaration; `/repo claim <name>` writes the claim request file into the mounted shared checkout (claims work from ro-mounted sessions BECAUSE the request file lives in the rw signals sidecar — `/repos/.signals/<name>/rw-request` — which is mounted rw for everyone from C1 per the Pinned names section; no mechanism decisions remain here).
- **AC13 – holder notification**: the statusLine command gains a segment — when this session holds the rw lock for a shared repo AND that repo's `rw-request` file exists (visible through `${VIBE_REPOS_DIR:-/repos}/.signals/<name>/`), append ` · ⚠ rw:<requesting-project>`. The `\a` bell is BEST-EFFORT/experimental: statusLine stdout re-renders per repaint and may sanitise control bytes — emit it, but no test asserts terminal behaviour; MANUAL-TESTS observes it, and if it rings per-repaint or is stripped, drop the byte in the same commit without failing the AC. The existing `test_vibe_statusline` exact-output fixtures (smoke-test.py ~6470, from the statusLine commit ee5cb81) must remain byte-identical when no shared repos/signals exist (segment absent); new fixtures cover request-present/absent via `VIBE_REPOS_DIR` pointed at a temp tree. Identifying "this session holds the lock": the launcher exports `VIBE_PROJECT_NAME` (basename of the host project dir) via remoteEnv; the statusLine compares it to the lock meta's `project=`.
- **AC14 – CLAUDE.md fragment**: `devcontainer/claude-md/shared-repos.md`, synced/gated by install-claude-extras ON the manifest being non-empty (brain2.md gating precedent). Content: how to read the manifest; every `/repos/<name>` is a separate proprietary (or at least separately-licensed) repo — NEVER copy code from `/repos/*` into `/workspace`; consume shared code only via the project's declared interface/feature-flag seam, never import internals; ro mounts mean commit/push happen in the shared repo's own vibe project, not here; `/repo claim` etiquette.
- **AC15 – gates** as AC7 (fragment shape test mirrors existing fragment tests; command file shape test mirrors vss.md tests; statusLine fixtures per AC13).

## Cycle 4 — per-repo credentials + invariant amendment (slices 5, 7 remainder)

- **AC16 – credential helper multi-token**: `devcontainer/credential-helper.sh` upgraded: still answers only `get` on https/github.com; parses `path` (present because of AC17) as `owner/repo(.git)`; if it equals the project's repo (new remoteEnv `GITHUB_REPO_SLUG` exported by the launcher) → serve `$GITHUB_TOKEN`; else if it matches a configured shared slug → serve that repo's `VIBE_SHARED_TOKEN_*` env var; else → serve NOTHING (git fails loud). **No fallback path may ever serve `$GITHUB_TOKEN` for a non-project repo** — the never-widen guarantee, and the thing the Tester must attack hardest (path missing, path with .git suffix, path with subdir suffix e.g. `owner/repo/info/refs`, case differences, a slug that sanitises to a colliding env name, a trailing-slash-only path, traversal segments like `owner/../other/repo`, and embedded-newline/injection strings in any credential-protocol stdin field).
- **AC17 – useHttpPath**: `credential.useHttpPath true` is set in-container at the same place the credential helper itself is configured today (find it — Dockerfile gitconfig or setup-git.sh — and set it at the same scope so it applies to /workspace AND /repos/*). Without it git never sends `path`, so AC16's helper must ALSO handle the no-path case: no path → serve `$GITHUB_TOKEN` ONLY if no shared repos are configured (compat), else serve nothing and let git fail loud (a mis-set useHttpPath must not silently widen).
- **AC18 – token plumbing**: this is NEW plumbing, not reuse — `render_devcontainer_with_mounts` today touches only `mounts`/`runArgs`, and devcontainer.json's `remoteEnv` is a fixed dict. The per-repo exports MUST happen in the launcher's top-level scope beside the existing `export GITHUB_TOKEN` (vibe ~1660s) — NEVER inside `_build_override_config`, which is invoked via command substitution where exports die with the subshell (the comment at vibe:1691-1696 documents this exact trap for learning_load; cite it in the code comment). The override-config generator then adds the matching `"VIBE_SHARED_TOKEN_X": "${localEnv:VIBE_SHARED_TOKEN_X}"` remoteEnv entries (extending render_devcontainer_with_mounts or a sibling to merge a remoteEnv object). Absent token → the mount still works (ro reads of a local checkout need no token) but `vibe repos list` and the header report the missing token as the BROKEN-state reason only for rw intents and fetch attempts; pin: missing token is a WARNING, not a mount-blocker.
- **AC19 – invariant text**: `/workspace/CLAUDE.md § Invariants` line "Fine-grained PATs are scoped to **one repo**..." amended to: each PAT stays single-repo; a container's blast radius = exactly the repos announced in its launch header, each via its own token; never a multi-repo token. README's security section updated to match.
- **AC20 – security-review**: the security-review agent reviews the C4 diff (credential-helper.sh, token plumbing, remoteEnv) before commit; its verdict is recorded in the session file and any CONFIRMED finding blocks the commit until fixed. (C1 got an advisory pass over the registry/mount injection surface.)
- **AC21 – gates**: code-check; smoke for AC16-AC18 (credential helper is directly executable — feed it `get` stdin fixtures with env combinations and assert exact username/password output or silence, including every never-widen attack case in AC16); MANUAL-TESTS section: end-to-end push to a shared repo with its own PAT, push to project repo unchanged, push to an undeclared repo fails; CHANGELOG; `vibe --version`-adjacent docs if any list features.

## Out of scope

- Cross-machine lock coordination (locks referee one Mac; contributors coordinate via git).
- Packaging/versioned-artifact consumption of the shared repo (release-time concern, explicitly deferred).
- Any change to guard-bash.sh, guard-fs.sh, init-firewall.sh, permission lists, or forceLoginMethod.
- Live mount-mode changes (impossible under Docker; handoff = exit + relaunch by design).
- Auto-cloning a missing shared repo from GitHub (registry points at an existing local checkout; cloning is the user's step — `vibe repos add` may PRINT the clone command but must not run it).
- Multi-host/NFS lock semantics; submodule integration; any UI beyond header/statusLine/CLI text.
