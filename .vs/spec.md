# Spec тАФ task_027: dotted-repo-name detection fix + brain2 vibe-operation note

## Task summary

Two defects surfaced 2026-07-21 by a real session (andeye_com). (A) `detect_github_repo`'s regex pins the repo segment to `[^/.]+`, so a repo whose NAME contains dots (`andeyePro/andeye.com`) never parses тАФ vibe wrongly claims "No GitHub repo found" and offers to create a new repo, on every launch. (B) In-container Claudes know nothing about the host-side vibe CLI, so when a launcher issue appears they guess (that session invented a nonexistent "path registry") or punt ("run `vibe --help` yourself"). Fix: (A) parse `owner/repo` allowing dots, stripping `.git` in shell; (B) the launcher maintains an auto-refreshed `vibe --help` note in the user's brain2 (`<brain2>/meta/vibe-operation.md`), and a new shipped CLAUDE.md fragment points in-container Claudes at it with a give-exact-commands discipline.

## Pinned contract

### A тАФ `detect_github_repo`

- New algorithm, replacing the current single regex: take `url=$(git remote get-url origin)` as today; strip ONE trailing `/` (`url="${url%/}"`); match `[[ "$url" =~ github\.com[:/]([^/]+/[^/]+)$ ]]`; let `slug="${BASH_REMATCH[1]}"`; strip ONE trailing `.git` (`slug="${slug%.git}"`); return the slug (echo, rc 0) ONLY if `is_valid_repo_slug "$slug"` passes; otherwise return 1. No other behaviour change; non-github and no-remote still return 1.
- Consequences pinned as correct and unchanged: `save_token`/`lookup_token` key literally (dots fine); credential-helper `_sanitise_slug` maps non-alphanumerics to `_` (dots fine); a repo literally named `foo.git` arrives as `foo.git.git` in the URL and resolves to `foo.git` (single strip тАФ never a loop).

### B тАФ brain2 ops note + fragment

- New sourceable launcher function `refresh_brain2_vibe_note`, ALWAYS returns 0 тАФ and the fail-soft mechanism is pinned, not just the outcome (an unguarded write into an unwritable dir under `set -euo pipefail` kills the whole sourced shell): EVERY filesystem operation in the function is individually errexit-guarded with stderr silenced тАФ `mkdir -p "$p/meta" 2>/dev/null || return 0`; `[ -w "$p/meta" ] || return 0`; content written to a temp file then moved, each step `2>/dev/null || return 0` (or the equivalent `if !` form). Belt-and-braces: the launch-path call site is written `refresh_brain2_vibe_note || true` (same call-context errexit suppression the task_026 `setup_token` fix pinned). Resolve `p=$(_brain2_source)`; return 0 silently when empty (covers `VIBE_BRAIN2_PATH=off`) or when `$p` is not an existing directory (never create the brain2 root; mirrors the mount-gating rule). Otherwise the guarded `mkdir -p "$p/meta"`; target file `$p/meta/vibe-operation.md`, created with the single heading line `# vibe тАФ CLI operation (auto-maintained)` if absent. Maintain a managed block at the END of the file, markers exactly `<!-- >>> vibe-cli-help (auto, do not edit) >>>` and `<!-- <<< vibe-cli-help <<< -->` (distinct from existing vibe markers); strip-and-reappend using the same awk idiom as `install_claude_md_fragments` (preserve ALL content outside the markers; trim trailing blank lines; idempotent тАФ two consecutive runs leave one block, byte-identical). Block body: a line `Generated by vibe <version> at <UTC ISO8601>` (version via `_vibe_version`; timestamp via `date -u`), then the full `vibe --help` text inside a ```text fence. Capture mechanism (sourcing-safe тАФ `usage()` greps `"$0"`, which is `bash` when vibe is sourced by a smoke test, so calling it is FORBIDDEN here): factor a new `_usage_text()` holding usage()'s two emission steps but grepping `"$VIBE_REPO_DIR/vibe"` (BASH_SOURCE-derived, sourcing-safe) instead of `"$0"`, with no `exit`; `usage()` becomes exactly `_usage_text` + `exit 0` (help output byte-identical); the note writer captures `help_text=$(_usage_text)`.
- Call site: one guarded line in the launch path (host-side, before container launch, after config/token setup тАФ placement flexible within that window), unconditional per launch. Subcommand paths (`pat`, `learn`, `repos`, `audit`, `--help`) do NOT refresh the note.
- New fragment `devcontainer/claude-md/vibe-cli.md` (auto-collected by the existing sorted-glob sync in `install_claude_md_fragments` тАФ no manifest edit exists or is needed). Content requirements (prose, concise, in the style of sibling fragments): for host-side vibe launcher matters (PAT setup/rotation, repo detection/registration, rebuilds, flags), the source of truth is `/brain2/meta/vibe-operation.md` when mounted тАФ it holds the CURRENT installed `vibe --help`, refreshed every launch; if brain2 is not mounted, ask the user to type `! vibe --help` so the output lands in the conversation; ALWAYS answer with the exact command to run and state which side (Mac host vs container) it runs on; NEVER answer with "go find the right command yourself" or equivalent. Must contain the literal strings `vibe-operation.md`, `! vibe --help`, and `exact command`; plus one line noting that prose OUTSIDE the note's managed block is ordinary brain2 content under the normal brain2 trust discipline (only the managed block is machine-generated).
